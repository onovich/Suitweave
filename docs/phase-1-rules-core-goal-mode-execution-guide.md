## 0. 直接给执行者的 Goal Prompt

在 `D:\WebProjects\Suitweave` 中执行 Phase 1“规则内核”。建立纯 TypeScript、确定性、可重放且不依赖 React/DOM/平台 API 的游戏领域核心：冻结最小 `GameState / Command / DomainEvent / Ruleset` 契约，实现三盘模型、四向同色连通域、覆盖规则、德扑/21 点/三消计分、完成与评级、seed RNG，并以规则测试和架构门禁证明正确性。严格按本指南逐轮执行；每轮必须完成 Debug 自检、架构自检、相关验证、精确提交与推送，推送成功后才可进入下一轮。使用 `$donextgoal` 执行，不扩大到 Phase 2 UI/完整回合循环。

## 2. 本阶段要完成什么

- `src/domain/` 的无副作用领域模型、品牌类型和值对象。
- 版本化 `Ruleset`，覆盖 7x7、24 回合、抽 6 用 3、四墨色、2-5 数字有效组等默认规则。
- 棋盘/格子三层状态：数字与花色、墨色、皇冠/小标记；锁定状态作为棋盘级约束。
- 可注入 seed RNG 与确定性随机序列；相同 seed 与命令产生相同结果。
- 四方向同墨连通域、数字数量分类、落单/过载判定。
- 覆盖与基础落牌的纯领域转换；非法动作返回可辨识失败，不部分修改状态。
- 三套独立计分器，完整覆盖 GDD Phase 1 所需计分表。
- 完成检测、盘提交前资格、结算惩罚、总评级计算。
- 单元测试、边界测试、属性/不变量测试和确定性回归夹具。
- 公共领域 API 的简明说明及 GDD 规则追踪表。

## 3. 本阶段不做什么

- 不实现抽 6 用 3 的完整回合应用用例、React 交互或拖拽预览。
- 不实现皇冠/灵感奖励流程、功能牌、存档适配器、遥测、音频或 Electron。
- 不进行玩法平衡改值，不引入角色、遗物、形状牌、数独盘或真实三消掉落。
- 不让 domain 依赖 React、DOM、Node、Electron、存储、网络或全局随机数。
- 不为了未来扩展建立万能框架；只抽象已存在两种以上实现的概念。

## 4. 每轮固定工作流

每轮回复必须包含：

- 本轮目标与完成内容。
- Debug 自检及定位到的具体层。
- 架构自检。
- 已运行验证命令与结果。
- commit hash、远端分支及 push 结果。
- 下一轮目标。
- 是否消耗缓冲轮；若是，说明原因。

推进门禁：验证失败不得提交或进入下一轮；提交失败或 push 失败不得进入下一轮。使用项目 `GitFlow` 包装器或等价的精确路径提交流程，不暂存无关文件。

每轮最低验证为相关测试、`pnpm typecheck`、`pnpm arch:check`。影响公共契约或结束一个工作块时运行完整 `pnpm validate`。

## 5. Debug 与架构自检

### Debug 自检

- 当前变化能否由最小棋盘夹具、牌组夹具或命令重放解释？
- 失败能否定位到模型、连通算法、规则转换、具体计分器、完成判定或 RNG？
- 是否覆盖成功、失败、空棋盘、落单、过载、爆点、不兼容规则版本等状态？
- 随机状态是否能通过 seed 和命令序列完整重放？
- 状态改变是否满足“非法动作零修改”和输入不变性？

### 架构自检

- `GameState` 与版本化 `Ruleset` 是否仍为唯一事实来源？
- 计分器是否只读取领域值，未复制连通检测或 UI 语义？
- `Command`、`DomainEvent`、规则判定与评分职责是否分离？
- domain 是否保持平台无关且无循环依赖？
- 是否避免提前拉入 Phase 2-6 范围？
- 是否保持无关文件、生成物和用户改动不变？

## 6. 分轮安排

### Round 1：冻结最小领域契约

建立目录、品牌类型、枚举、`Ruleset v1`、`GameState / Board / Cell / Card / Command / DomainEvent` 最小定义和测试构造器。产出契约说明。不得实现 UI 或应用层状态容器。

PASS：类型严格；非法结构不可轻易构造；默认 ruleset 精确对应 GDD 附录；完整 `pnpm validate` 通过。

### Round 2：连通域与棋盘分类

实现四方向同墨连通域、空墨桥、数字计数、落单、有效规模候选、过载分类。用 1x1、边角、断桥、同色相邻、异色相邻、0/1/2/5/6 数字等最小夹具覆盖。

PASS：每个有墨格恰属一个域；遍历顺序不改变结果；无对角连接；不变量测试通过。

### Round 3：落牌、覆盖与确定性 RNG

实现数字牌正面、背面仅墨色、纯墨色、万能墨色选择的领域转换；实现锁定拒绝、非法目标零修改；引入 seed RNG 与固定序列测试。

PASS：覆盖矩阵覆盖 GDD 6.2；同 seed 重放完全一致；不得使用 `Math.random()`。

### Round 4：三套计分器

分别实现德扑、21 点、三消计分器。德扑含 2-5 张最高牌型和 A 高低顺；21 点含 A 1/11、两张黑杰克、五龙、爆点；三消按 GDD 表只取最高模式。

允许执行者在公共评分输入契约冻结后，将三个计分器测试矩阵分给独立 worker；只有主执行者可修改公共契约，并负责合并与交叉复核。

PASS：表驱动测试覆盖全部计分行、平分优先级和无效边界；计分器不执行连通遍历。

### Round 5：有效组、完成、提交资格与评级

把连通分类与各盘计分器组合成统一分析结果；实现所有数字有墨、2-5 数字、无爆点/过载/落单的完成资格；实现未完成惩罚、盘完成奖励输入和 D-SS 评级纯函数。提交锁定可只实现领域状态转换，不实现奖励应用用例。

PASS：三盘规则分别验证；空棋盘不可提交；评级同时满足完成盘数门槛与分数门槛。

### Round 6：领域门面、重放与规则追踪

建立窄领域 API，统一返回状态与事件；补命令序列重放、预览可复用的纯执行入口和 GDD 规则追踪文档。预览不得另写规则，只允许调用同一转换函数处理临时状态。

PASS：相同初态/seed/命令日志产生字节语义等价结果；公共 API 无平台类型；完整 `pnpm validate` 通过。

### Round 7：缓冲修复轮

只处理前六轮暴露的规则遗漏、类型缺口、跨计分器不一致、架构违规、测试不稳定或文档差异。若无需修复，不制造新功能；记录“未消耗”，直接进入 Round 8。

PASS：所有已知缺陷有回归测试；无延期项被偷偷纳入。

### Round 8：最终验证与交接

执行完整验证、固定 seed 重放、架构审计和 Git 清洁检查；更新 roadmap 的 Phase 1 状态或新增完成报告，但不得规划或实现 Phase 2。

PASS：`pnpm validate` 全绿；零依赖违规；规则追踪无未知缺口；工作区干净；最终提交已推送 `origin/main`；向 planner 返回报告并请求 `$checkandgoal` 验收。

## 7. 总体 PASS 标准

- 三盘模型、连通、覆盖、三计分器、完成/评级、seed RNG 全部实现。
- 规则核心为纯 TypeScript，domain 无平台依赖、循环和 UI 规则复制。
- 关键规则具有表驱动或不变量测试；相同 seed 与命令可重放。
- `pnpm validate`、固定 seed 回归和 `git diff --check` 全部通过。
- 每轮均有独立 commit 且成功推送；最终 `main` 与 `origin/main` 同步。
- Phase 2 范围未提前进入。
