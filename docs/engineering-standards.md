# 工程规范与防漂移工作流

## 强制规范

- 产品名仅用“织花牌 / Suitweave”；旧名只能出现在历史来源说明中。
- 领域类型不使用裸 `string`/`number` 表达 BoardId、CellId、Score 等关键概念；公开函数必须有明确输入输出。
- 禁止 `any`、非空断言和静默 catch；边界输入先校验再进入 domain。
- 函数单一职责，复杂度 <=12、默认不超过 80 行；组件不包含计分和规则分支。
- 每项规则用 GDD 章节或 ADR 追踪；魔法数字集中到版本化 ruleset。
- commit 必须聚焦；改变行为必须同时提交测试与文档更新。

## Definition of Done

每个功能完成前必须：

1. 写清验收例、失败例和平台影响。
2. 在 domain/application 层先实现规则与测试，再接 UI。
3. 执行 `pnpm validate`：lint、typecheck、test、架构检查、生产构建全部通过。
4. 手动验证键盘与触摸主路径；涉及关键局面时保存固定 seed 回归样例。
5. 自检依赖方向、是否复制规则、是否污染平台边界、是否需要 ADR/存档迁移。

## 评审清单

- 规则：预览与执行是否同源？随机是否可重放？边界条件是否覆盖？
- 架构：domain 是否纯净？跨层 import 是否只朝内？是否形成新循环或万能模块？
- UI：三盘状态是否同时可知？无颜色信息是否仍可理解？焦点和触控是否可用？
- 平台：浏览器 API 是否被适配器包裹？Electron 威胁面是否扩大？
- 运维：测试失败信息是否可定位？文档、规则版本和存档版本是否一致？

## 变更控制

- 架构、规则语义、存档格式、多平台策略变化必须新增 `docs/adr/NNNN-*.md`。
- CI 与本地使用同一 `pnpm validate`，禁止只修 CI 脚本绕过失败。
- 每个 Phase 结束做一次架构审计：依赖图、循环、公共 API、测试覆盖盲区、ADR 与现实差异。
